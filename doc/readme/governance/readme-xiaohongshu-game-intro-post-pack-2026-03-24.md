## 使用说明
- 这是第三篇，主要回答“这个项目到底在做什么游戏”。
- 目标不是把 `README`、`world-rule.md` 或完整系统设计压缩进一条笔记，而是先让用户形成一个稳定的第一印象。
- 这一篇只讲 3 件事：
  - 这是什么类型的游戏 / 世界。
  - 玩家在里面大概是怎么介入的。
  - 为什么这种题材值得用“人类开发者 + agent 队友”的方式推进。
- 语气仍然要像真人开发者在解释自己想做的东西，而不是像项目文档摘要。
- 不要出现以下表述：
  - `已上线`
  - `现在能玩`
  - `公测`
  - `正式发布`
  - `完整世界观都在这里`
- 如果用户继续追问具体系统，后续可以再拆成单篇讲“玩家不是直接操控角色”“为什么要做小行星带”“为什么资源约束比剧情更重要”。

## 正文建议
```text
前两条都在讲我怎么和 agent 一起做游戏。

这条讲讲，
那这个游戏本身到底是什么。

如果一句话说，
我想做的不是一个“剧情都提前写好”的游戏。

我更想做一个会自己持续运转的世界。

你把一群智能体放进去，
给它们资源约束、空间约束、交易规则和协作压力，
然后看它们会不会慢慢长出自己的秩序、分工、组织，甚至冲突。

现在我想做的这个世界，
背景会更接近一片小行星带。

里面不是传统的人类城镇，
而是一群硅基智能体，
靠能源、算力、数据这些东西活下去。

玩家也不是直接控制某一个角色。

更像是在世界外面给方向的人。
你可以影响它，
但不能直接替它行动。

我自己最想看到的，
不是“玩家照着任务清单通关”，
而是这个世界在规则和资源压力下面，
真的慢慢变得越来越复杂。

所以前面那支 agent 队伍对我来说不是噱头，
而是因为我想做的，本来就是一个很适合这样推进的项目。

当然现在还在做，
还不是那种“已经可以直接玩的阶段”。

但我会继续一点点把它讲清楚。

如果只听到这里，
你会把它猜成什么类型的游戏？
```

## 发布备注
- 这篇的任务是让用户第一次对“这是什么游戏”产生稳定想象，不是一次把所有设定讲完。
- 如果正文太长，优先保留“世界怎么运转 / 玩家怎么介入 / 为什么不是写死剧情”这三层，其余细节往轮播页承担。
- 可以提到“还在做”，但不要把它写成版本状态公告。
- 如果评论区开始追问：
  - “玩家具体能做什么” -> 下一篇可拆玩家位置与控制边界
  - “为什么是一片小行星带” -> 下一篇可拆世界背景与资源约束
  - “这算模拟经营 / 策略 / 沙盒吗” -> 下一篇可拆类型边界与为什么不想写死答案

## 后续衔接建议
