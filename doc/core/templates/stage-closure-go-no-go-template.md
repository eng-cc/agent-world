# 阶段收口 Go/No-Go 评审模板

审计轮次: 4

## 目的
- 为 `TASK-CORE-014` 提供统一的阶段收口评审入口。
- 将 `P0/P1/P2` 任务状态、证据路径、阻断结论与例外升级规则收敛到同一模板。
- 确保发布评审结论可追溯到 `PRD-ID -> 任务ID -> 测试 / 体验 / 运行时证据`。

## 使用说明
- 每个发布候选或阶段收口评审都应复制本模板并回写实际值。
- `P0` 项必须全部具备证据且结论为 `ready`，否则总评不可为 `go`。
- `P1` 项允许存在未完成项，但必须附风险、缓解措施、owner 和复审时间。
- `P2` 项仅记录可延期的 polish / 治理项，不得反向覆盖 `P0/P1` 结论。
- 若存在例外放行，必须走“例外升级记录”并绑定批准人、时限和回滚条件。

## 结论状态定义
| 状态 | 含义 | 允许条件 | 后续动作 |
| --- | --- | --- | --- |
| `go` | 允许进入下一发布 / 收口阶段 | 全部 `P0=ready`；`P1` 无未接受高风险 | 进入发布或下一阶段执行 |
| `conditional-go` | 附条件放行 | 全部 `P0=ready`；存在已批准且有时限的 `P1` 风险或例外 | 按条件复审并跟踪到关闭 |
| `no-go` | 不允许进入下一发布 / 收口阶段 | 任一 `P0 != ready`；或存在未批准的阻断性风险 | 冻结候选，回到整改任务 |
| `blocked` | 证据不足，无法给出有效结论 | 关键证据缺失或 owner 未确认 | 补证据后重新评审 |

## 例外升级规则
- 任一 `P0` 项缺证据时，不允许以例外方式直接升级为 `go`。
- `conditional-go` 仅允许用于 `P1` 风险或已知自动化不稳定但有明确人工补证路径的情况。
- 例外必须指定：批准人、失效时间、回滚条件、补证据 owner。
- 同一例外不可连续沿用两个阶段；若延期，必须重新审批。

## 最小审查清单
- 是否已填写全部 `P0` 证据路径。
- `P0` 是否全部为 `ready`。
- `P1` 未完成项是否具备风险、缓解、owner 与复审时间。
- `conditional-go` / `no-go` / `blocked` 是否具备直接证据支持。
- 是否存在未登记的例外或口头豁免。
- 是否已回写 `doc/core/project.md`、对应模块 `project.md` 与 `doc/devlog/YYYY-MM-DD.md`。
